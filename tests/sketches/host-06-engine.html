Deep dive · the brain behind every seat

Seating Intelligence Engine

One pure, explainable, tunable module that answers "what's the best table for this party, right now — without hurting the rest of the night?" It weighs a dozen signals, learns your real turn-times, looks ahead at the whole book, and shows its work — and a manager can re-shape its judgement with sliders, no code.

Explainable · every score has a "why" Learns turn-times from history Manager-tunable policy
Design principle. A seating engine only gets used if operators trust it and can steer it. So two rules govern everything below: (1) hard constraints are never violated — the engine physically can't suggest a table that doesn't fit or is spoken for; (2) every soft judgement is transparent and tunable — the host sees the reasons, the manager owns the weights.

signals What it weighs

Signals split cleanly into hard (a filter — violate one and the table is out) and soft (a score — traded off against each other by policy weights). That split is what keeps it both safe and smart.

◆ Hard constraints — must pass

↔Capacity fitseats ≥ party (a 2-top can't hold 4)
◷Free for the full turnfree-window ≥ expected turn + reset buffer, else combinable
♿Accessibilitywheelchair / high-chair / step-free when the guest needs it
⛔Availabilitynot out-of-service, not held for a VIP, not already seated
⊕Join feasibilitylarge party → only adjacent, combinable tables qualify

◆ Soft signals — scored & traded off

▣Right-sizereward seats == party; penalise seating 2 at an 8-top (wasted covers)
◷Runway comfortmore clear time after the turn = safer, higher score
★Guest preferenceVIP · usual table · window/booth · zone · quiet — from CRM
⧗Pacingavoid stacking seatings in one 15-min bucket (kitchen surge)
◈Yield / inventoryprotect big tables & prime windows for known demand
⚖Section balancespread covers evenly across server stations

learns Real turn-times, not a flat 90 minutes

The single biggest lever on accuracy is how long a party actually stays. The engine learns it per party-size × daypart × weekday from every real seatedAt → closedAt, with a confidence band, and shrinks to sensible priors when a cell is thin. Cold-start works day one; it sharpens every service.

look-ahead Optimise the night, not just the seat

Greedy "best table for this party" is short-sighted — it can leave the book un-seatable later. The engine scores the marginal cost of each choice against the rest of the evening's demand (bookings + walk-in forecast), penalising fragmentation and any move that would block a known party.

closes the loop It gets better because you use it

Every suggestion → accept / override(reason) → outcome (actual turn, walk-out, complaint) is telemetry. That drives three things: turn-time learning, a weekly weight-tuning nudge ("hosts override 'yield' 40% of the time on slow nights — lower it?"), and an acceptance-rate dashboard so you can see the engine earning trust before you let it drive.

TracksuggestionId → shown rank, chosen table, override reason, realised turn, section load at seat.
Tunerolling acceptance rate per signal → surfaced as tuning suggestions in the config panel (never auto-changed without approval).
Provedashboard: acceptance %, avg covers/night vs baseline, predicted-vs-actual turn error, walk-in wait accuracy.

contract Pure, versioned, testable

suggest()suggest(party, at, prefs, policy, book) → Ranked[]. Pure function. Each result: {tableId|join, score, breakdown, reasons[], confidence, expectedTurn, freesAt}. Deterministic → snapshot-testable.
simulate()simulate(book, policy) → {utilization, predictedWaits, atRisk[]}. Runs the engine over the whole reservation book pre-service to forecast pressure and flag un-seatable parties early.
learnTurn()updateTurnModel(session) on every close → the (party×daypart×dow) estimate + confidence. Bounded, incremental.
policyVersioned config per location: weights, caps, buffers, hold rules, presets. Lives beside loyalty/growth settings; editable in admin. Backward-compatible defaults.

delivery Ship the intelligence in three versions

v1
rules

Explainable rules engine

Hard filters + weighted soft score with static, sensible weights + preset presets. Turn-time = size-based defaults. Ships the "smart seat" + walk-in guard with full explainability. Enough to be trusted.

v2
tunable

Manager policy + pacing/yield/section

The config panel (weights, caps, holds, presets), pacing/yield/section signals, and the pre-service simulate() forecast. Now it's yours to shape.

v3
learning

Turn-time learning + look-ahead + trust loop

Learned turn-times with confidence, the look-ahead fragmentation optimiser, override-driven weight tuning, and the acceptance/utilisation dashboard. It compounds.

Why this is professional-grade

  • Hard constraints are inviolable — safe by construction.
  • Every decision is explained & every override audited.
  • Pure & deterministic → unit + snapshot testable.
  • Cold-starts on defaults, sharpens with data.
  • Policy is versioned config, not code — per location.

    • What to watch

  • Turn-time learning needs a few weeks of closes to beat defaults.
  • Look-ahead optimiser adds compute — keep it heuristic for live, exact for pre-service.
  • Too many weights can confuse — lead with presets, hide the sliders under "advanced."
  • Section balance needs server-station data (Phase-3 alongside TableSession).
