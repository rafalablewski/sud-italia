Concept 5 of 5 · The recommendation

Three lenses, one shared truth

Don't pick one view — the door, the manager and the server each need a different one. Instead, build Concept 4's Table-Session as the shared spine and render the other three as lenses over it. Then ship it in three phases, each valuable on its own, the first with zero schema change.

Recommended path Fits the existing Lens IA Phase 1 ships in days
The core decision. The mistake would be choosing Floor vs. Timeline vs. Arrivals. They aren't competitors — they're the same data seen by different roles. The system already speaks in lenses (Service = Floor · Slots · Dispatch · Book). Add a shared TableSession underneath and make seating one primitive; the three views become three tabs that can never disagree.

deltas Exactly what changes

Phase 1+ nothing in the DB. New: freeWindowMin() helper, a reservation overlay on CoreFloor, and POST /floor/seat {reservationId, tableId} (sets seated + opens check). Book & Floor already share the reservations store.
Phase 2+ walk-in as a lightweight hold (ad-hoc reservation, source:"walk-in", no name) so the floor counts it. New: suggestTables() + the + Walk-in guard UI. Reuses Book's conflict primitive.
Phase 3+ TableSession entity + Waitlist; FloorTable.status becomes derived; endpoints /table-sessions, /floor/state, /suggest, /waitlist. Add Timeline & Arrivals lenses to the Service surface.
