The Table-Session model

The other four concepts are views. This is the foundation they all want underneath them: one entity that unifies "a booking," "a seated party," and "a walk-in" into a single lifecycle — so every surface reads the same truth about who occupies what, when.

The layer under everything else One entity: TableSession + an assignment engine
Why the break exists at all. Today a Reservation (with tableId + status) and a FloorTable (with status) each hold half the truth, and nothing owns the join. "Who is at T5 right now, and until when?" has no single record to answer it. A walk-in has no record at all. We fix the model, and every view gets correct for free.

model One record owns "party × table × span"

Introduce TableSession — the single occupancy record. A reservation is a future intent; on arrival it's realised into a session (source booking). A walk-in creates a session directly (source walk-in). The table's live status is derived from its sessions, never set by hand.

Seating a booking and seating a walk-in converge on the same active session — that's what makes both problems disappear: Book learns the table filled (via reservationId), and the floor learns a walk-in exists (a session with no reservation).

engine The assignment / free-window engine

One pure module every view calls. It answers "what can I safely seat this party on, right now?" by scoring each table on fit, free-window (won't collide with the next reservation) and preference — and refuses the ones that aren't genuinely free.

// pure, testable — the brain behind "smart seat" everywhere
function suggestTables(party, at, prefs) {
  return tables
    .map(t => ({ t,
      fits:   t.seats >= party && t.seats <= party + 2,   // right-sized
      freeFor: freeWindowMin(t, at),                  // until next reservation/session
      zoneHit: prefs.zone ? t.zone === prefs.zone : true,
    }))
    .map(x => ({ ...x, ok: x.fits && x.freeFor >= TURN_MIN(party),
                 score: score(x) }))          // fit + runway + zone − oversize
    .sort((a,b) => b.score - a.score);
}
// freeWindowMin = minutes from `at` until this table's next hold
//   (active session end, OR an upcoming reservation start). 0 ⇒ occupied.

trade-offs Honest read

Strengths

  • Fixes the root cause, not a symptom — every view becomes correct.
  • Walk-ins finally have a record; reporting & covers get accurate.
  • One derived floor-state feed = no more two-systems drift.
  • Assignment engine is reusable, pure, unit-testable.

    • Limits

  • Biggest lift — new entity, migration, backfill, endpoints.
  • Not a UI on its own; needs a view (Concepts 1–3) on top.
  • Deriving table status changes existing Floor write paths.
  • Over-engineered if you only ever run one small room.
