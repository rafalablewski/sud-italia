# Core v2 — the separated rebuild

A ground-up, 1:1 rebuild of the Core operational suite onto the
**core-suite mockup** (`public/mockups/core-suite/`), living beside the
current `/core/*` surfaces under `/core-v2/*` so we can swap once it
reaches parity. Started because the v1 shell diverged from the mockup
(it shipped a two-row header; the mockup is a left **sidebar**), and the
fixes never converged — v2 starts clean.

## Doctrine — a truly separate entity

Core v2 inherits **nothing** from Admin or Homepage. It does not import
their theme CSS, tokens, or shell. The only shared code is the **data
layer** (API routes + `src/lib/*` + the location / toast / currency
providers) — behaviour, not styling — so every surface stays wired to
real data.

- **Theme:** `src/app/core-v2/styles/` — a self-contained 1:1 port of the
  mockup's `system.css`.
  - `core.css` — the system (tokens, sidebar shell, primitives, extended
    kit, dialogs). **Every rule is scoped under `.corev2`** and the design
    tokens live on `.corev2` (not `:root`), so the theme can never leak
    into `/admin` or the storefront. Fonts come from the v2 layout's own
    next/font vars (`--cv2-display` / `--cv2-ui` / `--cv2-mono`), mapped to
    `--display` / `--ui` / `--mono`.
  - `pos.css` — the POS surface styles (port of `pos.html`'s `<style>`),
    imported by the POS page. Each surface gets its own file the same way.
- **Layout:** `src/app/core-v2/layout.tsx` — loads only `core.css` + the
  v2 fonts and the data/behaviour providers (`AdminLocationProvider`,
  `AdminCurrencyGuard`, `ToastProvider`). No admin theme import.
- **Shell:** `src/core-v2/shell/CoreShellV2.tsx` — the mockup's left
  **sidebar** chrome shared by every surface: brand, a **Core** nav group
  (POS · KDS · Guest Engagement) + an **Operations** group (Menu · Floor),
  a user foot fed by `/api/admin/me`, and a breadcrumb `.topbar` whose
  trailing slot holds each surface's own controls.

  ```
  <CoreShellV2 active="pos" crumb="POS" topbar={controls}>{body}</CoreShellV2>
  ```

  `SurfacePlaceholder` renders this chrome with a "next in the rebuild"
  body for surfaces not yet ported.

## Scoping & portals

Because the whole theme is scoped under `.corev2`, any node rendered
through `createPortal(node, document.body)` (the fullscreen kiosk,
dialogs) **must** be wrapped in `<div className="corev2">` so the scoped
styles + tokens reach it (CLAUDE.md rule #4). The `.corev2` root is a
fixed full-viewport canvas; a portaled `.corev2` resets that so it acts
only as a token scope.

POS reuses the exact endpoints v1 does — `/api/admin/pos/tabs`,
`/api/admin/pos/orders`, `/api/admin/floor/tables`,
`/api/admin/pace/steering` — and the same `@/lib/upsell` +
`@/lib/pos-coursing` logic; it is a re-skin onto the mockup markup, not a
new backend.

## Authority

Code wins over docs — when they disagree, fix the doc. When the mockup
and a v1 behaviour disagree, the **mockup** wins (that is the point of
v2); when functionality and the mockup disagree, keep the functionality
and find the mockup's nearest affordance.
